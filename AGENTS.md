# Core patterns

Imports use the `@/` alias (repo root). Scripts: `dev`, `build`, `lint`, `typecheck`, `test` (vitest), `format` (prettier), `migrate`.

- Always inline logic at its single call site instead of extracting one-off helpers by default.

## Errors as values (`utils/result.ts`, `utils/errors.ts`)

- Never throw across the server boundary — return a `Result<D, E>`. Check `result.ok`, then read `result.val` or `result.err`.
- Return success with `ok(val)`. For errors, use the typed constructors (`notFound`, `forbidden`, `invalidInput`, `bug`, ...) rather than calling `err` directly. `invalidInput` drives form field errors; other types surface as dialogs.
- Wrap throwing code with `catchBug(origin, fn)` to convert exceptions into a `BUG` result.

## Data layer (`data/`)

- Three layers, wired by dependency injection: `repo.ts` (Kysely queries) → service factory (`createXService(repo)`, holds business logic + validation) → `data/index.ts` (`"use server"`, instantiates services with `db` and re-exports actions).
- Factories are `createXRepository(db)` / `createXService(repo)`; export the type via `ReturnType<typeof ...>`.
- Always import actions and types from `@/data` (never from `repo.ts`/service files directly). These are server actions, so client components can call them directly — no API route or fetch needed.

## Database (`db/`, Kysely + Postgres via `pg`)

- Query through the Kysely `DbClient` from `@/infra/db`; use the query builder, not raw SQL.
- Connection comes from `DATABASE_URL` (`config.ts`). The app client (`db/index.ts`) reads it via `config`; `db/migrator.ts` reads `process.env.DATABASE_URL` directly so it stays self-contained for the Docker migration stage — keep it free of app imports.
- Migrations live in `db/migrations/` (Kysely `up`/`down` files). Manage them with `npm run migrate`: `create <name>`, `up`, `down`, `latest`, `reapply`, `reset`, `list`. Pass `--yes`/`-y` to skip confirmation on destructive commands. Applying migrations regenerates `db/types.d.ts`.
- `db/types.d.ts` is generated by kysely-codegen — do not edit by hand.

## Models & types

- Validate/parse with Zod (`z.object`, `z.infer`).
- Use nominal types for ids: `Nominal<string, "UserId">` (`utils/typing.ts`).

## UI (Next.js App Router)

- Server components are `async` and call data actions directly; mark client components with `"use client"`.
- Mutations use `<form action={...}>` + `useActionState`, then `router.refresh()` from `next/navigation`.
- UI components come from shadcn (in `components/`, see `components.json`), built on `@base-ui/react` primitives. Add new ones with the `shadcn` CLI rather than hand-rolling. Style with Tailwind v4 via `cn()` (`utils/styling.ts`) and `cva` variants. Reuse `utils/formatting.ts` helpers.

## Config & tests

- Env via `@t3-oss/env-nextjs` + Zod in `config.ts`; import the default `config`.
- Vitest with globals (no imports for `describe`/`it`/`expect`); follow the `// Given / // When / // Then` comment style.
